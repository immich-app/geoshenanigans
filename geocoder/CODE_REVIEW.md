# Geocoder code-quality review (2026-06-17)

Parallel multi-agent review of the builder (C++) + server (Rust), for the refactor pass on `refactor/geocoder-quality`.
Hard invariants every refactor must preserve: **(1) determinism** — chained build from identical PBF reproduces a byte-identical index (same-PBF planet patch ~243 B, verify 26/26); **(2) Nominatim parity** — 500-city query results unchanged. HIGH/MEDIUM findings only (low-severity + full text were in the agent output).

## Full PR-branch review round (2026-07) — 47 verified findings shipped

A second, branch-wide review (8 areas, adversarially verified) landed five

continent-filter race and the GCPATCH v3 gate), three behaviour fixes

patch sizes, 327K-coord grid identical, planet-scope byte diffs exactly
attributable to the three intended behaviour changes.

## Refactors applied in this pass (validated)

Every change below was validated against the hard invariants before commit. "byte-identical" means a same-binary build was diffed file-by-file against a pre-change baseline (`cmp -s` over every output `.bin`); "planet" includes TIGER + wikidata sitelinks. The two builder refactors are determinism-critical, so both were proven byte-identical at oceania **and** planet scale — byte-identical output is the strongest possible proof a refactor changed no behaviour (it also implies Nominatim parity is untouched).

- **Test infrastructure (was: none).** Added a dependency-free C++ unit harness (`tests/test_framework.h` + `tests/*.cpp`, a `geocoder-tests` CMake target) — 99 tests / 391 checks over the builder's pure logic (id_allocator, postcode, geometry, string_pool, ring_assembly, interpolation, patch_format). Added Rust `#[cfg(test)]` coverage to the server (6 → 49 tests) over the query-path pure helpers. No production code touched by the test commits.
- **`cell_item_less` hoist** (`16eec17`). The cell→item index sort tiebreak (cell_id, then item_id) was an identical inline lambda copy-pasted in 7 ordering/cell-build sites; hoisted to one `cell_item_less()` in `types.h`. _Validated byte-identical: oceania 131/131, planet 131/131._
- **`classify_poi` extraction** (`48a47d3`). The ~350-line captureless POI-classifier lambda was lifted out of `main()` to a file-scope `static` free function (reusing the existing `PoiClassification` struct from `types.h`, which the inline copy was shadowing). Addresses the "classify_poi lambda" HIGH item below. _Validated byte-identical: oceania 131/131, planet 131/131._
- **`float_bits` hoist + re-inlined `cell_item_less` cleanup** (`05b22b9`). Lifted the IEEE-754 total-order `float_bits` lat/lng tiebreak (used by all 6 deterministic-ordering sort comparators) from a `main()` lambda to a file-scope inline function, and replaced the two POI/addr parent-street `lower_bound` probes that re-inlined the `cell_item_less` comparator with the hoisted helper. _Validated byte-identical: oceania 131/131, planet 131/131._
- **Byte-safe code-quality polish** (`0b654c3`). Replaced the scattered raw `0x7FFFFFFF`/`0x80000000` admin cell-id literals with the existing `ID_MASK`/`INTERIOR_FLAG` constants (dropping local `ADMIN_*` duplicates; left the `float_bits` IEEE-754 sign-bit literal — same value, different meaning); named the FNV-1a basis/prime (`FNV1A_OFFSET_BASIS`/`FNV1A_PRIME`); and made the TIGER/postcode loaders check the `system()` (tar/gunzip) return code and bail instead of silently proceeding. Addresses the three MEDIUM items so marked below. _Validated byte-identical: oceania 131/131, planet 131/131._
- **`decode_polygon_verts` F32 alignment UB fix** (`8190e6f`, server). The encoding-4 branch cast a 2-aligned byte pointer to `*const NodeCoord` (needs 4-byte alignment) — undefined behaviour. Replaced with per-field unaligned `f32::from_le_bytes` reads (bit-identical on little-endian) + added a test. Risk-free: the builder only ever emits encodings 0/1/2/3, so encoding 4 is a dead/legacy reader branch — cannot affect any current query result. Addresses the first "Suspected pre-existing bugs" item below.

### Remaining structural refactors — deliberately left for human-supervised application

These are high-value but either invasive or judgment-dependent, so they are documented (below) rather than auto-applied. All are byte-validatable with the same oceania→planet `cmp` methodology, which makes them *safe to attempt* — the reason to hold them is that they need a human to drive the design / confirm intent, not that they're unverifiable:

- **`main()` decomposition** (HIGH) — **IN PROGRESS (`c82c706`, `687565b`, `d54969c`, `c9fc761`, `338dbc5`, `b48e82e`).** Seam chosen: free functions taking `ParsedData& data` + a new `BuildConfig` struct (config from argv), with the shared running timers passed by reference where needed. Extracted 9 self-contained post-parse phases (`compute_admin_parent_chain`, `compute_way_parent_polygons`, `inherit_cdp_postcodes`, `backfill_addr_point_parent_streets`, `compute_place_node_containment`, `link_places_by_name`, `compute_poi_parent_streets`, `compute_poi_parent_postcode`, `compute_poi_parent_admin`) + hoisted `parallel_sort_and_build` to file scope (`CellItem`→`CellItemPair`). **Then restructured the S2 section** (`338dbc5`, `b48e82e`): lifted it into `compute_s2_and_poi_cells(data, cfg, _pt&, _cpu&)` which owns the section timer `_s2t`, and sub-extracted `compute_s2_cells_ways_interp` + `compute_poi_s2_cells` — `compute_s2_and_poi_cells` is now a 36-line orchestrator calling the 6 S2/POI phase functions. **`main()` is now ~3,673 lines (was ~5,154; −29%).** Every step moved verbatim via an asserted extraction script and validated byte-identical oceania **and** planet (131/131). **Remaining inline (bigger/riskier — need explicit go-ahead):** the PBF parse passes (Pass 1/2/2b, admin/POI relation assembly — thread-local-accumulator heavy), the deterministic-ordering pass (byte-layout-defining — highest risk), and the write pass.
- **Parent-linking / containment dedup** (HIGH, below). The 4 sweeps (admin-containment, way-parent, POI-parent, centroid-postcode) share a skeleton but have likely-intentional per-site differences (tie-breaks, candidate sources). A naive unification would either fail byte-identical or need so many flags it isn't cleaner — a human should confirm which differences are essential before unifying. (Note: the PIP *primitive* is already factored into `point_in_polygon`/`poly_contains`; this is about the surrounding loops.)
- Smaller captureless lambdas (`place_rank`, `float_bits`, `addr_equal`, `quality_dir_name`, `area`, `rebuild_map_parallel`) — extracting them is a readability wash relative to the per-change validation cost; fold them in opportunistically alongside the main() decomposition. (`parallel_sort_and_build` and `float_bits` are now done; `place_rank` moved into `link_places_by_name`.)

### Rural US postcode gap — FIXED (2026-07): TIGER per-segment ZIP sidecar

The architectural fix described below has been implemented: the builder
keeps each TIGER segment's ZIP (`interp_postcode_ids`, written as
`interp_postcodes.bin`, u32 string offset per interp way, NO_DATA for OSM
interpolations, absent on non-TIGER builds), the patch tools carry it as
patch fid INTERP_POSTCODES (GCPATCH v3), and the server consults it as
postcode tier 1c — gated on the street winning the primary, the segment
lying within ~100m, and a normalised token overlap between the segment's
TIGER street name and the winning way's name (Nominatim's
`_find_tiger_number_for_street` parent join, approximated by name). The
fragile nearest-centroid tier-2 pipeline was left completely untouched.

#### Original analysis (historical)


A live-Nominatim comparison over a rural-US grid sample found **~52% of our "no postcode" addressed rural points are a real divergence** — Nominatim returns a 5-digit ZIP (e.g. 77713, 13630, 03570, 32325) where we return none; the other ~48% are genuine wilderness where both return nothing. These are plain 5-digit ZIPs, **not** the ZIP+4 issue.

Root cause (traced to source): modern Nominatim `reverse.py` has **no nearest-centroid postcode lookup** — the reverse postcode comes from the **matched object's own row**, and for US rural roads it is stored **per TIGER segment** (`t.c.postcode`), so it's returned with no distance limit. Our model instead has (tier 0) the addr's own `addr:postcode` (rural roads lack it), (tier 1) `way_postcodes` from `boundary=postal_code` PIP (the US largely has no such polygons), and (tier 2) nearest postcode **centroid** capped at 5.5km over the query cell + immediate L10 neighbors. In rural US a ZIP's single averaged centroid is far from the query and outside both the radius and the searched cells → no postcode. The "matches Nominatim's get_nearest_postcode" comment on tier 2 is **stale** (it modeled an older Nominatim).

The faithful fix is **builder-side**: attach the TIGER segment's ZIP to the road/addr at build time (so reverse returns it directly with no distance limit), mirroring Nominatim. This is an architectural change — most likely an `InterpWay`/record field (→ binary-format + patch-wire-format change, determinism-sensitive) or a build-time nearest-TIGER-ZIP attachment to streets/addr — and needs its own design + full parity cycle. Widening the tier-2 radius is **not** a faithful substitute: the cell search only covers adjacent L10 cells, and a larger radius risks returning a neighbouring ZIP. **Surfaced to the user as the highest-impact postcode follow-up; deferred from this pass.**
